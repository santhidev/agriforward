# ADR-0001: ใช้ระบบเงินสดเต็มจำนวนใน Escrow (ไม่มีเครดิต/แบ่งงวดใน MVP)

## สถานะ

Accepted

## บริบท (Context)

ในตลาดเกษตร B2B มูลค่าดีลต่อครั้งสูง (หลักหมื่นถึงแสนบาท) มีความเสี่ยงจากทั้งสองฝ่าย:
- Buyer กลัวโอนเงินแล้วไม่ได้ของ
- Seller กลัวส่งของแล้วไม่ได้เงิน

PRD v2.0 เสนอระบบเครดิต (Credit Line) และการแบ่งจ่ายเป็นงวด (Installment) โดยแพลตฟอร์มเป็นผู้ให้เครดิตและรับความเสี่ยงเอง

## การตัดสินใจ (Decision)

**MVP จะใช้โมเดลเงินสดเต็มจำนวน (100% upfront cash) ผ่านระบบ Escrow ของแพลตฟอร์ม**

- Buyer จ่ายเงิน 100% ก่อนรับสินค้า
- แพลตฟอร์มถือเงินไว้ใน escrow
- ไม่มีระบบเครดิต (Credit Line)
- ไม่มีการแบ่งจ่ายเป็นงวด (Installment)
- ไม่มี commission จากดีล

## ผลที่ตามมา (Consequences)

### ดี (Pros)
- **ลดความเสี่ยงทางกฎหมาย**: ไม่เข้าข่ายสถาบันการเงิน/สินเชื่อส่วนบุคคล ไม่ต้องขอใบอนุญาตจาก ธปท.
- **ลดความซับซ้อนทางเทคนิค**: ไม่ต้องสร้างระบบ credit scoring, credit limit, dunning, collection
- **สภาพคล่องของแพลตฟอร์ม**: ไม่ต้องใช้ทุนสำรองจ่ายให้ Seller ก่อน
- **ชัดเจน**: Buyer รู้ว่าต้องจ่ายเท่าไร Seller รู้ว่าจะได้เท่าไร

### ไม่ดี (Cons)
- **Barrier สำหรับ Buyer**: Buyer ที่สภาพคล่องไม่พออาจไม่สามารถซื้อได้
- **ยอดดีลอาจน้อยกว่า**: ไม่มี leverage จากเครดิตทำให้เกิดดีลน้อยลง
- **ต้องพึ่ง Escrow**: แพลตฟอร์มต้องมีความน่าเชื่อถือสูงมาก เพราะถือเงินเต็มจำนวน
- **ไม่มีรายได้จากดอกเบี้ย/เครดิต**: สูญเสียช่องทาง monetization หนึ่งช่องทาง

## ทางเลือกอื่นที่พิจารณา (Alternatives Considered)

| ทางเลือก | เหตุผลที่ไม่เลือก |
|----------|------------------|
| **Credit Line + แบ่งงวด 3 งวด** | เข้าข่าย nano-finance ต้องขอใบอนุญาต ธปท. ความเสี่ยงทางกฎหมายสูง |
| **Buyer จ่ายตรงให้ Seller (ไม่มี escrow)** | ไม่แก้ปัญหาความเชื่อถือ แพลตฟอร์มไม่มี value proposition |
| **แบ่งงวดแบบไม่มีเครดิต (มัดจำ 30% + 70% ตอน POD)** | ยังต้องมีระบบติดตามงวด ซับซ้อนกว่าใน MVP |

## หมายเหตุ

- Post-MVP อาจพิจารณาเพิ่มระบบเครดิตเมื่อมีใบอนุญาตที่ถูกต้อง
- การตัดสินใจนี้ส่งผลต่อการออกแบบ Wallet, Payment, และ State Machine ทั้งหมด
