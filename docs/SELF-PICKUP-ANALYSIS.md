# วิเคราะห์กรณี Self-Pickup — "แพลตฟอร์มเปิดกว้างและฟรี"

> เอกสารนี้วิเคราะห์กรณี Self-Pickup ในบริบทที่แพลตฟอร์มต้องการเป็น **platform ที่เปิดกว้างและฟรี** โดยบริการขนส่งเป็นเพียง **อ็อฟชั่นเสริม (optional paid add-on)**

---

## 1. หลักการหลัก (Core Philosophy)

```
┌─────────────────────────────────────────────────────┐
│  "ใช้แพลตฟอร์มฟรี — จ่ายเฉพาะถ้าอยากให้เราส่ง"      │
└─────────────────────────────────────────────────────┘
```

| บริการ | ค่าใช้จ่าย | สถานะ |
|--------|-----------|-------|
| โพสต์ PO | ฟรี | หลัก |
| เสนอราคา (Offer) | ฟรี (แค่มัดจำ 5%) | หลัก |
| ตรวจ QC | ฟรี | หลัก |
| รับสินค้าที่ Hub (Self-Pickup) | **ฟรี** ✅ | **DEFAULT** |
| ส่งถึงบ้าน (Platform Logistics) | เสียเงิน 💰 | อ็อฟชั่น |

---

## 2. ผลกระทบต่อ State Machine

### 2.1 โฟลว์หลัก (Default = Self-Pickup)

```
[Default Path — ฟรี]
Draft → Open → AwaitingSellerConfirm → PaymentPending → Contract 
→ QualityPending → ReadyForPickup → Fulfilled (Buyer มารับที่ Hub)

[Optional Path — เสียเงิน]
Draft → Open → AwaitingSellerConfirm → PaymentPending → Contract 
→ QualityPending → InTransit → Fulfilled (แพลตฟอร์มส่ง)
```

### 2.2 ความแตกต่างระหว่าง 2 paths

| จุด | Self-Pickup (ฟรี) | Platform Logistics (เสียเงิน) |
|------|-------------------|-------------------------------|
| **เงินที่จ่าย** | 100% ของสินค้า | 100% ของสินค้า + ค่าขนส่ง |
| **State หลัง QC** | ReadyForPickup | InTransit |
| **Auto-confirm** | 7 วัน | 48 ชม. |
| **รายได้แพลตฟอร์ม** | 0 (จากดีลนี้) | ค่าขนส่ง |
| **Seller ได้เงิน** | 100% (หลัง POD) | 100% (หลัง POD) |
| **Buyer ทำอะไร** | มารับที่ Hub | รอที่บ้าน |

### 2.3 คำถามสำคัญ: ถ้า Buyer เลือก Self-Pickup แต่ไม่มารับ?

| ทางเลือก | ผล | แนะนำ |
|----------|-----|-------|
| A) Auto-confirm หลัง 7 วัน | เงินโอนให้ Seller ดีลจบ | ✅ **ใช้นี้** |
| B) ยกเลิกดีล คืนเงิน Buyer | แต่สินค้าอยู่ที่ Hub แล้ว ใครรับผิดชอบ? | ❌ ซับซ้อน |
| C) เก็บค่าเก็บรักษา | เป็นรายได้เพิ่ม แต่ทำให้ "ฟรี" ดูไม่ฟรี | ❌ ขัด philosophy |

**แนะนำ A:** Auto-confirm 7 วัน เงินโอนให้ Seller สินค้าอยู่ที่ Hub ถือว่า Buyer มารับแล้ว (หรือปล่อยทิ้ง — แต่ในทางปฏิบัติอาจต้องมีนโยบายเก็บสินค้าที่ Hub กี่วัน)

## 3. ผลกระทบต่อ UI/UX

### 3.1 หน้าสร้าง PO (PoNew.razor)

**แนะนำการออกแบบ:**

```
┌─────────────────────────────────────────┐
│  วิธีรับสินค้า                          │
│                                         │
│  ◉ มารับที่ Hub เอง (ฟรี)    ← DEFAULT  │
│    ที่อยู่ Hub: 123 ถนน..., กรุงเทพฯ    │
│                                         │
│  ○ ส่งถึงที่อยู่ของฉัน (+฿500)          │
│    [เลือกที่อยู่...]                     │
│                                         │
└─────────────────────────────────────────┘
```

**หลักการ:**
- Self-Pickup เป็น **default** (pre-selected radio button)
- แสดงที่อยู่ Hub ชัดเจน (พร้อมแผนที่/เบอร์โทร)
- Platform Logistics เป็น **secondary option** ที่ต้องกดเลือกเอง
- ถ้าเลือก logistics → แสดงราคาทันที (real-time calculation)

### 3.2 หน้า PO Detail (PoDetail.razor)

**สถานะที่ต้องแสดง:**

| State | ข้อความที่แสดง (Self-Pickup) | ข้อความที่แสดง (Logistics) |
|-------|------------------------------|----------------------------|
| QualityPending | "สินค้าอยู่ระหว่างตรวจสอบคุณภาพ กรุณารอสักครู่" | เหมือนกัน |
| ReadyForPickup | "✅ QC ผ่านแล้ว! มารับสินค้าได้ที่ Hub" | — |
| InTransit | — | "🚚 สินค้ากำลังจัดส่ง รอรับสินค้าได้เลย" |

### 3.3 การแจ้งเตือน (Notifications)

**Event ที่ต้องแจ้งเตือนเพิ่มสำหรับ Self-Pickup:**

1. **QC Passed + Self-Pickup** → "สินค้าพร้อมรับแล้ว! มารับได้ที่ Hub ตั้งแต่วันนี้"
2. **ใกล้ครบ 7 วัน** → "⏰ เหลืออีก 2 วัน หากไม่มารับระบบจะ auto-confirm ให้อัตโนมัติ"
3. **Auto-confirm** → "ดีลสำเร็จ (auto-confirm) เนื่องจากไม่ได้มารับภายใน 7 วัน"

## 4. ผลกระทบต่อ Seller

### 4.1 Seller ไม่รู้ว่า Buyer เลือกอะไร?

**สถานการณ์:**
- Seller เห็นแค่ว่า "ต้องส่งมอบที่ Hub"
- Seller ไม่จำเป็นต้องรู้ว่า Buyer จะมารับเองหรือให้แพลตฟอร์มส่ง
- Seller ทำหน้าที่เหมือนกันทุกกรณี: ส่งมอบที่ Hub + รอ QC

**ข้อดี:**
- ลดความซับซ้อนฝั่ง Seller
- Seller ไม่ต้องจัดการขนส่งเองเลย
- ไม่ต้องกังวลเรื่อง buyer อยู่ไกล/ใกล้

### 4.2 ถ้า QC Failed — ผลต่อ Seller

| กรณี | ผล | เงินมัดจำ |
|-------|-----|-----------|
| สินค้าที่ส่งมาไม่ตรงสเปค | ดีลยกเลิก | คืนเต็ม |
| สินค้าเสียหายระหว่างขนส่งมา Hub | ดีลยกเลิก | คืนเต็ม |

> หมายเหตุ: ถ้าสินค้าเสียหายระหว่างขนส่งจาก Seller → Hub ใครรับผิดชอบ? นี่คือช่องโหว่ที่ต้องระบุใน Terms of Service

## 5. ผลกระทบต่อแพลตฟอร์ม (Platform Perspective)

### 5.1 รายได้จาก Logistics จะน้อย?

ถ้า Self-Pickup เป็น default → อาจมีคนเลือก Platform Logistics น้อย

**วิธี promote โดยไม่บังคับ:**

| กลยุทธ์ | ตัวอย่าง | ผล |
|---------|----------|-----|
| **ลดราคาโปรโมชั่น** | "ส่งฟรี! 3 ดีลแรก" | ช่วงแรกให้ลองใช้ |
| **ขาย convenience** | "ไม่ต้องขับรถ เราส่งถึงบ้าน" | เน้นประหยัดเวลา |
| **Volume discount** | "ส่งฟรีเมื่อซื้อครบ 50,000" | กระตุ้นยอด |
| **B2B targeting** | Buyer รายใหญ่ (โรงงาน) อาจไม่มีเวลามารับ | ธุรกิจใช้ logistics มากกว่า |

### 5.2 ถ้าคนส่วนใหญ่เลือก Self-Pickup — แพลตฟอร์มเสียเปรียบไหม?

**ไม่เสียเปรียบ เพราะ:**
1. **Volume** — ถ้า platform ฟรี → มีคนใช้เยอะ → network effect
2. **Data** — ได้ข้อมูลดีล ราคา จำนวน → วิเคราะห์ตลาดได้
3. **Trust** — ตรวจ QC ฟรี = สร้าง trust ในตลาด
4. **Upsell opportunity** — คนที่ลอง Self-Pickup ครั้งแรก ครั้งต่อไปอาจใช้ logistics

### 5.3 ต้นทุน Hub

ถ้าคนมารับเยอะ → ต้องมีพื้นที่ Hub ที่เพียงพอ + พนักงานรับมอบ

**คำถาม:**
- Hub อยู่ที่ไหน? (กรุงเทพฯ / ต่างจังหวัด / หลายสาขา?)
- เปิดกี่โมง? (ถ้าเป็นเกษตรกร อาจมารับเช้ามาก/เย็นมาก)
- มีลานจอดรถใหญ่พอไหม? (รถกระบะ/สิบล้อ)

## 6. กรณี Edge Cases

### 6.1 Buyer เลือก Self-Pickup แต่พอถึง Hub บ่นว่าสินค้าไม่ตรง

**ปัญหา:** QC ผ่านแล้ว แต่ Buyer ไม่พอใจ

**ทางเลือก:**
- A) ไม่รับสินค้า → ยกเลิกดีล → คืนเงิน Buyer (แต่ seller ส่งของถูกต้องแล้ว ใครผิด?)
- B) รับไปก่อน → ยกข้อพิพาทภายหลัง → Frozen
- C) มี "acceptance QC" อีกรอบตอน Buyer มารับ

**แนะนำ C:** ให้ Buyer ตรวจสอบก่อนรับที่ Hub (visual inspection) ถ้าไม่ตรง → ปฏิเสธรับ → ดีลยกเลิก → คืนเงิน (แต่ต้องมีกล้องวงจรปิด/พนักงานยืนยัน)

### 6.2 Buyer เลือก Self-Pickup แล้วมารับแต่ลืมพกบัตรประชาชน/OTP

**แนะนำ:**
- ต้องมี **pickup code** (QR หรือ 6-digit code) ส่งไปใน email/app
- หรือเช็คชื่อกับบัตรประชาชน
- ถ้าให้คนอื่นมารับแทน → ต้องแจ้งล่วงหน้า + ระบุชื่อคนรับแทน

### 6.3 Seller ส่งของมา Hub แต่ Hub เต็ม/ปิด

**แนะนำ:**
- ต้องมี **appointment system** สำหรับ Seller ส่งมอบ (ไม่ใช่ walk-in)
- หรือมี **drop-off time window** (เช่น จันทร์-ศุกร์ 9:00-16:00)
- ถ้า Hub เต็ม → ระงับรับ PO ชั่วคราว หรือขยาย Hub

## 7. สรุปผลกระทบต่อ MVP

### ✅ ต้องมีใน MVP

| ฟีเจอร์ | เหตุผล |
|---------|--------|
| Self-Pickup เป็น default | ตาม philosophy "ฟรีและเปิดกว้าง" |
| แสดงที่อยู่ Hub ชัดเจน | Buyer ต้องรู้ว่าไปรับที่ไหน |
| Pickup code / QR | ยืนยันตัวตนเมื่อมารับ |
| Auto-confirm 7 วัน | ป้องกัน buyer ไม่มารับ ดีลค้าง |
| QC ที่ Hub | รับประกันคุณภาพ ไม่ว่า buyer เลือกอะไร |

### ❌ ไม่ต้องมีใน MVP

| ฟีเจอร์ | เหตุผล |
|---------|--------|
| Appointment system สำหรับ pickup | ใช้ walk-in ก่อน หรือนัดผ่าน comment |
| ค่าเก็บรักษา (storage fee) | ขัด philosophy "ฟรี" |
| Multiple Hub locations | เริ่มจาก Hub เดียว |
| กล้องวงจรปิด/พยาน | ใช้ comment + dispute system |

## 8. คำถามสุดท้ายที่ต้องตัดสิน

### Q: ถ้า Buyer เลือก Self-Pickup แล้วเปลี่ยนใจอยากให้ส่ง?

| ทางเลือก | ผล | แนะนำ |
|----------|-----|-------|
| A) เปลี่ยนได้ถ้าสินค้ายังไม่ถึง Hub | ยอมรับ logistics fee เพิ่ม | ✅ ใช้ได้ |
| B) เปลี่ยนได้ถ้าอยู่ใน ReadyForPickup | ชำระ logistics fee + รอจัดส่ง | ซับซ้อน |
| C) ไม่เปลี่ยนได้เลย | ต้องสร้าง PO ใหม่ | ❌ ไม่ user-friendly |

**แนะนำ A:** เปลี่ยนได้จนกว่าสินค้าจะถึง Hub (หรือจนกว่าจะ QC ผ่าน) โดยชำระค่าขนส่งเพิ่ม
